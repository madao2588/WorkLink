# WorkLink 后端总体架构设计

## 1. 目标

当前 WorkLink 前端已经具备以下能力：

- 登录入口
- 消息列表与聊天详情
- 通讯录
- 工作台
- 审批列表
- 考勤状态
- 个人中心

后端设计目标不是只“能返回数据”，而是要满足以下要求：

- 可以直接对接现有 Flutter 前端
- 支持未来从 mock 平滑迁移到真实接口
- 支持消息实时推送
- 支持审批、考勤、个人中心等企业业务持续扩展
- 支持后续拆服务和多租户演进

## 2. 总体架构选型

建议第一阶段采用“模块化单体”架构。

### 2.1 技术建议

- API 协议：REST
- 实时通信：WebSocket
- 主数据库：PostgreSQL
- 缓存：Redis
- 对象存储：MinIO / OSS / COS
- 消息队列：RabbitMQ
- 日志追踪：ELK 或 Loki + Grafana
- 鉴权：JWT Access Token + Refresh Token

### 2.2 为什么先做模块化单体

- 你当前项目规模更适合快速联调和快速迭代
- 审批、考勤、个人中心都强依赖事务和关系数据
- 一开始就拆微服务会显著增加开发、部署和联调成本
- 聊天、通知、文件存储以后仍然可以单独拆出

### 2.3 后续可拆分方向

当用户量上升或实时业务明显增多后，可以把以下模块拆成独立服务：

- Chat Service
- Notification Service
- File Service
- Search Service

## 3. 业务模块划分

后端建议按以下模块划分：

### 3.1 Auth 认证中心

职责：

- 登录
- 登出
- token 刷新
- 账号状态校验
- 设备管理
- 登录审计

### 3.2 User & Org 用户与组织中心

职责：

- 用户资料
- 部门树
- 岗位
- 直属上级
- 通讯录搜索
- 用户在线状态展示基础信息

### 3.3 Chat 消息中心

职责：

- 会话列表
- 单聊消息
- 群聊扩展能力预留
- 已读未读状态
- 会话置顶与免打扰
- WebSocket 消息推送

### 3.4 Approval 审批中心

职责：

- 审批模板
- 审批单创建
- 审批节点流转
- 同意/拒绝/撤回
- 工作台待办聚合

### 3.5 Attendance 考勤中心

职责：

- 打卡
- 当日出勤状态
- 历史记录
- 异常状态
- 规则配置

### 3.6 Profile 个人中心

职责：

- 个人资料
- 设置
- 工资条
- 勋章
- 帮助反馈

### 3.7 Notification 通知中心

职责：

- 系统通知
- 审批提醒
- 公告提醒
- 聊天相关提示

## 4. 分层架构设计

每个模块统一按 4 层设计：

### 4.1 API 层

职责：

- 对外暴露 REST 接口或 WebSocket 入口
- 参数校验
- 统一响应结构
- 权限检查入口

### 4.2 Application 层

职责：

- 编排业务流程
- 管理事务
- 聚合多个领域对象
- 调用仓储与外部服务

### 4.3 Domain 层

职责：

- 表达核心业务规则
- 定义实体、值对象和领域服务
- 保持业务约束一致

### 4.4 Infrastructure 层

职责：

- 数据库访问
- Redis 缓存
- 消息队列
- 文件存储
- 第三方服务接入

## 5. 与当前 Flutter 前端的对接关系

Flutter 端现在已经开始采用 `feature -> repository -> provider -> screen` 的结构。

后端建议这样对应：

- Flutter `repository` 对应后端 API
- Flutter `provider` 负责页面状态
- 后端返回“可直接渲染”的摘要数据
- 复杂业务规则尽量留在后端

典型映射：

- `ConversationSummary` 对应消息列表摘要接口
- `ApprovalModel` 对应审批列表摘要接口
- `AttendanceProvider.status/time` 对应今日考勤状态接口
- `UserModel` 对应用户基础信息 DTO

## 6. 部署架构

第一阶段建议：

- Nginx / API Gateway
- WorkLink Backend App
- PostgreSQL
- Redis
- Object Storage
- RabbitMQ

### 6.1 部署链路

客户端 -> 网关 -> 后端应用 -> 数据库 / Redis / MQ / 存储

### 6.2 网关职责

- HTTPS 终止
- 鉴权透传
- 限流
- 请求日志
- 静态资源代理

## 7. 缓存设计

Redis 推荐缓存以下内容：

- Access Token 黑名单
- Refresh Token 状态
- 用户在线状态
- 热门通讯录搜索结果
- 会话摘要缓存
- 未读数缓存
- 工作台聚合卡片数据

缓存原则：

- 强一致事务数据以数据库为准
- 高频展示摘要适合缓存
- 工资条、审批详情等敏感数据谨慎缓存

## 8. 实时通信设计

### 8.1 REST 负责

- 拉列表
- 拉历史消息
- 拉详情
- 提交操作
- 分页

### 8.2 WebSocket 负责

- 新消息推送
- 已读回执
- 在线状态变化
- 会话摘要变化
- 系统通知到达

建议事件名：

- `message.new`
- `message.read`
- `conversation.updated`
- `user.presence.changed`
- `notification.new`

## 9. 安全设计

### 9.1 认证

- Access Token：短时有效
- Refresh Token：长时有效
- 支持设备维度登录会话管理

### 9.2 授权

- RBAC 角色权限模型
- 数据权限按本人、部门、管理员范围控制

### 9.3 敏感数据保护

重点保护对象：

- 工资条
- 登录设备
- 审批记录
- 账号安全操作日志

要求：

- 后端必须强校验权限
- 敏感接口必须写审计日志
- 文件下载要有访问权限和有效期

## 10. 可观测性

建议统一接入：

- 请求日志
- TraceId
- SQL 慢查询监控
- 错误日志聚合
- WebSocket 在线连接数监控
- MQ 消费积压监控

## 11. 推荐落地顺序

建议按下面顺序开发：

1. 认证与用户中心
2. 通讯录与组织结构
3. 聊天会话摘要与消息历史
4. 审批列表与审批操作
5. 考勤状态与打卡
6. 个人中心设置与工资条
7. 通知中心与 WebSocket

## 12. 一句话结论

最适合当前 WorkLink 的后端形态是：

基于 PostgreSQL、Redis、REST、WebSocket 的模块化单体架构，先满足当前 Flutter 前端联调，再逐步演进为可拆分服务的企业协同系统。
